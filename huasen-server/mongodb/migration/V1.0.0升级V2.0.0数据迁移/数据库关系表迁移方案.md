# 数据库关系表迁移方案

## 1. 背景

早期V1.0.0数据库设计中，实体间的关联关系以数组形式嵌入在文档内部，存在以下问题：

- 数据冗余：同一关系可能双向存储
- 查询效率低：关联查询需要解析JSON数组
- 维护困难：关系变更时需同步更新多处数据
- 扩展性差：无法为关系添加额外属性

## 2. 迁移目标

将嵌入式的关联关系迁移为独立的中间关系表，实现多对多关系的规范化设计。

## 3. 数据模型变更

### 3.1 journal ↔ column 关系

| 项目     | 旧设计                           | 新设计                  |
| -------- | -------------------------------- | ----------------------- |
| 存储位置 | `journal.columnStore` (JSON数组) | `journal2column` 关系表 |
| 字段格式 | `["columnId1", "columnId2"]`     | `{journalId, columnId}` |

**旧数据示例：**

```json
// journal 表
{
  "_id": "journal_001",
  "columnStore": "[\"column_001\", \"column_002\"]"
}
```

**新数据示例：**

```json
// journal2column 表
[
  { "journalId": "journal_001", "columnId": "column_001" },
  { "journalId": "journal_001", "columnId": "column_002" }
]
```

### 3.2 column ↔ site 关系

| 项目     | 旧设计                                                                | 新设计                      |
| -------- | --------------------------------------------------------------------- | --------------------------- |
| 存储位置 | `column.siteStore` (JSON数组)<br>`site.expand.columnStore` (双向冗余) | `column2site` 关系表        |
| 字段格式 | `["siteId1", "siteId2"]`                                              | `{columnId, siteId, order}` |

**说明：** 旧设计中 column.siteStore 与 site.expand.columnStore 为同一关系的双向存储，迁移时统一从 column.siteStore 读取，无需处理 site.expand.columnStore。

### 3.3 site ↔ pin 关系

| 项目     | 旧设计                       | 新设计                       |
| -------- | ---------------------------- | ---------------------------- |
| 存储位置 | `site.expand.pin` (名称数组) | `pin` 表 + `site2pin` 关系表 |
| 字段格式 | `["pin名称1", "pin名称2"]`   | `{siteId, pinId}`            |
| 特点     | 仅存储名称，无独立实体       | 独立实体表，支持扩展属性     |

**旧数据示例：**

```json
// site 表
{
  "_id": "site_001",
  "expand": "{\"pin\": [\"推荐\", \"热门\"], \"tag\": [\"技术\", \"教程\"]}"
}
```

**新数据示例：**

```json
// pin 表
[
  { "_id": "pin_001", "name": "推荐" },
  { "_id": "pin_002", "name": "热门" }
]

// site2pin 表
[
  { "siteId": "site_001", "pinId": "pin_001" },
  { "siteId": "site_001", "pinId": "pin_002" }
]
```

### 3.4 site ↔ tag 关系

| 项目     | 旧设计                       | 新设计                       |
| -------- | ---------------------------- | ---------------------------- |
| 存储位置 | `site.expand.tag` (名称数组) | `tag` 表 + `site2tag` 关系表 |
| 字段格式 | `["tag名称1", "tag名称2"]`   | `{siteId, tagId}`            |
| 特点     | 仅存储名称，无独立实体       | 独立实体表，支持扩展属性     |

## 4. 迁移脚本

| 序号 | 脚本文件                | 源数据                | 目标表             | 功能说明                         |
| ---- | ----------------------- | --------------------- | ------------------ | -------------------------------- |
| 1    | `journal2column.js`     | `journal.columnStore` | `journal2column`   | 解析数组，建立关系记录           |
| 2    | `column2site.js`        | `column.siteStore`    | `column2site`      | 解析数组，建立关系记录，保留顺序 |
| 3    | `site2pin.js`           | `site.expand.pin`     | `pin` + `site2pin` | 创建pin实体，建立关系记录        |
| 4    | `site2tag.js`           | `site.expand.tag`     | `tag` + `site2tag` | 创建tag实体，建立关系记录        |
| 5    | `cleanup-old-fields.js` | -                     | -                  | 清理旧字段，保持数据一致性       |

## 5. 执行步骤

### 5.1 前置检查

- [ ] 确认数据库备份已完成
- [ ] 确认目标关系表已创建（journal2column、column2site、site2pin、site2tag）
- [ ] 确认 pin、tag 表已创建

### 5.2 执行迁移

通过 `/app/runCode` 接口，按顺序执行以下脚本：

```
Step 1: 执行 journal2column.js   # 迁移 journal-column 关系
Step 2: 执行 column2site.js      # 迁移 column-site 关系
Step 3: 执行 site2pin.js         # 迁移 site-pin 关系 + 创建 pin 实体
Step 4: 执行 site2tag.js         # 迁移 site-tag 关系 + 创建 tag 实体
Step 5: 执行 cleanup-old-fields.js  # 清理旧字段（建议观察一段时间后执行）
```

### 5.3 验证结果

执行每个脚本后，检查返回的统计信息：

```
- totalJournals/totalColumns/totalSites: 参与迁移的记录总数
- parsedJournals/parsedColumns/parsedSites: 成功解析的记录数
- newRelations: 新建的关系记录数
- duplicateRelations: 已存在的关系数（跳过）
- invalidEntries: 解析失败的记录（需人工排查）
```

## 6. 注意事项

1. **幂等性**：迁移脚本（Step 1-4）可重复执行，已存在的关系会被跳过
2. **数据完整性**：迁移前会验证关联ID是否存在，无效引用会被过滤
3. **业务兼容**：迁移过程仅向新表写入数据，不修改原有表数据，不影响现有业务
4. **site.expand.columnStore**：该字段与 column.siteStore 为冗余存储，已在 column2site.js 中统一处理
5. **pin/tag 名称去重**：已存在的 pin/tag 不会重复创建
6. **清理时机**：建议在迁移完成后观察一段时间（如1-2周），确认业务运行正常后再执行清理脚本
7. **清理不可逆**：cleanup-old-fields.js 执行后 journal.columnStore 和 column.siteStore 字段将被删除，请确保已完成数据库备份

## 7. 回滚方案

如需回滚迁移结果：

```javascript
// 清空 journal2column 表
await req.working([{ schemaName: 'journal2column', methodName: 'deleteMany', payloads: [{}] }]);

// 清空 column2site 表
await req.working([{ schemaName: 'column2site', methodName: 'deleteMany', payloads: [{}] }]);

// 清空 site2pin 表
await req.working([{ schemaName: 'site2pin', methodName: 'deleteMany', payloads: [{}] }]);

// 清空 site2tag 表
await req.working([{ schemaName: 'site2tag', methodName: 'deleteMany', payloads: [{}] }]);

// 清空 pin 表（如需）
await req.working([{ schemaName: 'pin', methodName: 'deleteMany', payloads: [{}] }]);

// 清空 tag 表（如需）
await req.working([{ schemaName: 'tag', methodName: 'deleteMany', payloads: [{}] }]);
```

## 8. 旧数据清理

迁移完成并验证无误后，需清理旧数据字段以保持数据一致性。

### 8.1 清理内容

| 表      | 字段                 | 清理操作                      |
| ------- | -------------------- | ----------------------------- |
| journal | `columnStore`        | 删除字段                      |
| column  | `siteStore`          | 删除字段                      |
| site    | `expand.pin`         | 从 expand 对象中移除 pin 字段 |
| site    | `expand.tag`         | 从 expand 对象中移除 tag 字段 |
| site    | `expand.columnStore` | 从 expand 对象中移除该字段    |

### 8.2 清理脚本

| 脚本文件                | 功能说明                         |
| ----------------------- | -------------------------------- |
| `cleanup-old-fields.js` | 统一清理所有旧字段，返回清理统计 |

### 8.3 执行清理

**前置条件：**

- [ ] 确认迁移脚本（Step 1-4）已全部执行成功
- [ ] 确认新关系表数据完整性已验证
- [ ] 建议观察一段时间，确保业务运行正常

**执行方式：**

```
Step 5: 执行 cleanup-old-fields.js
```

### 8.4 清理结果验证

执行后检查返回的统计信息：

```
journal:
  - total: journal 记录总数
  - cleaned: columnStore 已删除的记录数
  - errors: 更新失败的记录

column:
  - total: column 记录总数
  - cleaned: siteStore 已删除的记录数
  - errors: 更新失败的记录

site:
  - total: site 记录总数
  - cleaned: expand 字段已更新的记录数
  - pinRemoved: 移除 pin 字段的次数
  - tagRemoved: 移除 tag 字段的次数
  - columnStoreRemoved: 移除 columnStore 字段的次数
  - errors: 更新失败的记录
```

### 8.5 清理后数据示例

**journal 表：**

```json
// 清理前
{
  "_id": "journal_001",
  "columnStore": "[\"column_001\", \"column_002\"]"
}

// 清理后（字段已删除）
{
  "_id": "journal_001"
}
```

**column 表：**

```json
// 清理前
{
  "_id": "column_001",
  "siteStore": "[\"site_001\", \"site_002\"]"
}

// 清理后（字段已删除）
{
  "_id": "column_001"
}
```

**site 表（保留 expand 中的其他字段）：**

```json
// 清理前
{
  "_id": "site_001",
  "expand": "{\"pin\": [\"推荐\"], \"tag\": [\"技术\"], \"columnStore\": [\"column_001\"], \"other\": \"value\"}"
}

// 清理后
{
  "_id": "site_001",
  "expand": "{\"other\": \"value\"}"
}
```
